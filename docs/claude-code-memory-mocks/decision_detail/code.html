Switch to SQLite WAL mode

Optimization of database write operations for concurrent local-first sessions and edge deployments.

database Database speed Performance bolt Low Latency
Context
Low write performance during concurrent sessions. Our current implementation encounters database lock contention when multiple local processes attempt to sync changes simultaneously, causing noticeable UI lag.
Decision

Enable Write-Ahead Logging (WAL) mode.

All database connections will be configured to utilize WAL instead of the default DELETE journaling method across all environment stages.

Rationale
psychology

Significant throughput improvement

WAL mode provides up to 3x throughput improvement by allowing readers to not block writers and vice versa. This provides significantly better concurrency handling for local development where multiple daemon processes might be accessing the store.

Alternatives Considered

Stay with DELETE mode

Rejected: Too slow for our high-frequency update requirements. Lock contention is unacceptable for modern UX.

Migrate to Postgres

Rejected: Too complex for local-first architecture and increases operational overhead significantly for minimal gains.

warning Consequences
  • arrow_right Small increase in disk usage due to WAL file persistence.
  • arrow_right Requires careful file handle management in multi-process environments.
